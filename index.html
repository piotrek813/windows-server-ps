Oczywiście samo wyświetlanie serwisów byłoby zupełnie niepotrzebne, jeśli nie moglibyśmy nic z nimi zrobić. Mamy więc kilka komend, które umożliwiają nam uruchomienie, zatrzymanie, bądź restart usługi.

Tylko teraz uwaga, bo to nie takie proste. Po wpisaniu której kolwiek z komend w większości przypadków możemy otrzymać komunikat podobny do tego:

Restart-Service : Service 'Klient DHCP (DHCP)' cannot be stopped due to the following error: Nie można utworzyć usługi WinHttpAutoProxySvc na komputerze '.'.
Stop-Service service_name Start-Service service_name Restart-Service service_name
Jak sobie z tym poradzić? Wbrew pozorom to dość proste. Błąd występuje, ponieważ usługa nie ma skonfigurowanego trybu uruchamiania, bądź ma go skonfigurowanego w nie taki sposób, jakiego oczekujemy.

Jak widzimy parametr “StartMode” ustawiony jest na „Auto”. W tym stanie serwer nie pozwoli nam zarządzać stanem uruchomienia usługi. Trzeba zmienić ustawienie na „Manual”.

Działa? Działa. Teraz możemy dowolnie zarządzać statusem uruchomienia usługi. Pamiętajmy jednak, że podany przykład jest dość dyskusyjny. W końcu DHCP powinno działać cały czas, tak więc ustawienie StartMode „Auto” jest zdecydowanie bardziej właściwe. Gdy skończymy się już bawić możemy je przywrócić w ten sam sposób, w jaki zmieniliśmy je na „Manual”.

Get-CimInstance win32_service | Where-Object Name -eq "dhcp" ProcessId Name StartMode State Status ExitCode --------- ---- --------- ----- ------ -------- 636 Dhcp Auto Running OK 0 Set-Service dhcp -StartupType manual Get-CimInstance win32_service | Where-Object Name -eq "dhcp" ProcessId Name StartMode State Status ExitCode --------- ---- --------- ----- ------ -------- 636 Dhcp Manual Running OK 0
Skrypty te mogą być bardzo użyteczne. Dajmy na to, że mamy jakiś mało wiarygodny program zewnętrzny, który może co jakiś czas się wyłączać. Piszemy więc prosty skrypt, którym od razu będziemy w stanie go uruchomić, jednocześnie sprawdzając jego status:

Get-Service service_name | Where {$_.status –eq 'Stopped'} | Start-Service

I voi’la! Należy tylko wywołać skrypt i możemy być pewni, że nasz program jest uruchomiony. A jeśli coś nie działa to cóż… wina administratora. Czyli nasza. Hmm… Wracamy do konfiguracji.
Jako ciekawostkę można dodać tylko, że dokładnie w ten sam sposób możemy zarządzać stanem uruchomienia komputerów i procesów w naszej domenie. Tak więc nudzący się admin ma całkiem spore pole do popisu, jeśli chodzi o żartowanie sobie z nielubianych przez niego pracowników.
Zarzązanie obiektami chyba najlepiej pokazać na przykładzie.
